# Инфраструктура Профессионального Перевода

## Цель

Профессиональный режим должен быть управляемым производственным контуром:
стоимость предсказуема, терминология переиспользуется, качество проверяется, а
история переводов сохраняется для аудита и повторного использования.

## Postgres

Postgres нужен как источник истины для данных, которые нельзя надёжно держать
только в `project.json`:

- проекты, пользователи, настройки качества и выбранные модели;
- запуски этапов, стоимость, токены, ошибки провайдеров;
- глоссарии, термины, версии терминов и ручные правки;
- кэш переводов сегментов и результатов `timing_fit`;
- история расчёта стоимости до запуска и фактической стоимости после запуска.

Минимальные таблицы следующего этапа:

- `projects`;
- `pipeline_runs`;
- `provider_usage`;
- `glossaries`;
- `glossary_terms`;
- `segment_translations`;
- `cost_estimates`.

## LightRAG

LightRAG (`lightrag.bigalexn8n.ru`) нужен для контекстной памяти проекта:

- хранить утверждённые переводы терминов и фраз;
- искать похожие сегменты из прошлых переводов;
- подмешивать релевантные термины в prompt перевода;
- собирать доменный контекст по длинным видео и сериям.

Интеграция должна идти через отдельный сервисный слой, а не напрямую из
pipeline-stage. Тогда CLI, API и будущие агенты смогут использовать одну и ту
же память.

## Автосоздание Глоссария

Планируемый процесс:

1. После транскрибации выделить кандидаты терминов: имена, бренды, продукты,
   технические термины, аббревиатуры.
2. Сгруппировать повторы и похожие написания.
3. Запросить LLM-кандидаты перевода с объяснением уверенности.
4. Проверить через LightRAG, были ли эти термины уже утверждены.
5. Сохранить черновик глоссария в Postgres.
6. Для любительского режима применять автоматически с осторожными правилами.
7. Для профессионального режима показывать человеку экран подтверждения перед
   запуском дорогих моделей.

## Расчёт Стоимости

Следующий слой расчётов должен считать:

- примерное число input/output токенов для перевода;
- число rewrite-попыток по сегментам, которые не помещаются в тайминг;
- стоимость TTS на следующих этапах;
- запас на fallback/повторные попытки;
- фактический расход после завершения по `usage` ответов провайдеров.

Для UI нужны две цифры: «оценка до запуска» и «факт после запуска».
